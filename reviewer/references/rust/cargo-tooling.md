# Cargo and Tooling

Use this reference for `Cargo.toml`, `Cargo.lock`, workspaces, features, build scripts, editions, MSRV, lints, profiles, dependencies, crate exports, and generated code.

## Core Stance

- Match the repo's Cargo workflow, workspace layout, edition, MSRV, lint policy, feature resolver, and dependency style.
- Treat `Cargo.lock` as generated by Cargo. Do not hand-edit it.
- Do not change broad workspace config to solve a local crate issue unless that is the requested change.
- Dependency and feature changes are public behavior. Keep them narrow and validate affected packages.
- Prefer existing dependencies and standard library APIs before adding new crates.
- Config changes need validation with the same commands CI or local scripts run.

## Workspaces and Packages

- In a workspace, know whether commands should run at root, with `--workspace`, with `-p`, or in a member directory.
- Respect `default-members`, virtual manifests, shared workspace dependencies, shared package metadata, and workspace lints.
- Add dependencies to the crate that uses them, not to a convenient root manifest unless the workspace owns dependencies centrally.
- Preserve crate boundaries. Do not reach into another crate's private modules by path.
- If changing public exports, check downstream crate imports in the workspace.

## Features

- Features should be additive. Enabling a feature should not remove behavior or break other feature combinations.
- Avoid feature names like `default`, `std2`, `new`, or placeholders. Use meaningful capability names.
- Be careful with default features on dependencies. Turning them on can pull in `std`, TLS, async runtimes, proc macros, or large transitive graphs.
- Test important feature combinations: default, no-default-features, all-features, and targeted feature sets when the crate supports them.
- Keep `std`/`alloc`/`no_std` boundaries intact when present.
- Feature-gated public API needs docs and tests for the enabled/disabled states.

## Dependencies

- Check license, maintenance, MSRV, feature defaults, transitive cost, and existing workspace versions before adding a dependency.
- Prefer stable, ecosystem-standard crates when the problem is non-trivial: serde, thiserror, anyhow, tokio, tracing, clap, tempfile, proptest, criterion, etc., according to repo convention.
- Avoid adding a crate for a tiny helper unless it prevents real risk.
- Do not expose dependency types in public APIs unless that dependency is part of the API contract.
- Keep dev-dependencies out of production feature paths.

## Editions, MSRV, and Lints

- Preserve `edition` and `rust-version` unless the task is an explicit migration.
- New syntax or APIs must fit the crate's MSRV.
- Use `cargo fmt` for formatting. Do not hand-format away from rustfmt.
- Run Clippy according to repo policy. Treat correctness lints seriously; avoid mass allow attributes.
- Add `#[allow]` only near the code that needs it and with a reason when the reason is not obvious.
- Do not disable warnings or lints globally to pass a local change.

## Build Scripts, Proc Macros, and Generated Code

- Build scripts should be deterministic, declare rerun conditions, and avoid hidden network or host-specific assumptions.
- Generated code should come from the generator command.
- Proc macros should keep diagnostics clear and avoid surprising generated API.
- Be careful with environment variables in build scripts; they affect reproducibility and cross-compilation.
- Keep platform-specific code behind cfgs with tests or compile checks when possible.

## Common Findings

- Dependency added to wrong crate or without lockfile update.
- Feature is not additive or breaks no-default-features.
- Public API exposes a dependency type accidentally.
- MSRV is broken by a newer language feature or std API.
- Clippy/rustc lint is allowed globally for a narrow issue.
- Build script depends on undeclared files or environment without `rerun-if-*`.
- Workspace-level config changed when one member crate should change.
- Generated code or lock file was manually edited.
